# Plan: Fix Footer Overlap with Foreground Clusters in Gallery

## Root Cause Analysis

**File**: `src/components/home/ClusterLayout.tsx`

The layout stacks three layers in the viewport:

| Layer | Position | Z-Index |
|---|---|---|
| Main Stage (cluster cards) | `absolute top-[16vh] md:top-[32vh]` | `z-10` |
| Footer (scrollable thumbnails) | `fixed bottom-6 md:bottom-10`, height `14vh`/`16vh` | `z-20` |
| Bottom-right buttons | `fixed bottom-6 right-6` | `z-[400]` |

### Problem 1: No bottom spacing on the main stage (line 291)

```txt
absolute top-[16vh] md:top-[32vh] left-0 w-full h-auto md:h-[44vh] 
overflow-y-auto overflow-x-hidden md:overflow-hidden custom-scrollbar z-10
```

- **Mobile** (`h-auto` + `overflow-y-auto`): Content grows to full height. Without `padding-bottom`, the last cluster cards scroll into/behind the `fixed` footer (z-20 > z-10).
- **Desktop** (`md:h-[44vh]` + `md:overflow-hidden`): Fixed height ends at `32vh + 44vh = 76vh`. Footer is `bottom-10` (≈4vh) with height `16vh`, so footer top ≈ `100 - 4 - 16 = 80vh`. There's a 4vh gap — not clipping but no safe margin either.

### Problem 2: Fixed height on desktop ignores footer presence

The `md:h-[44vh]` is a hardcoded value that doesn't account for the fixed footer at the bottom. On shorter viewports (laptops, tablets in landscape), the 44vh could be too large, causing the bottom of cluster cards to sit behind the footer.

### Problem 3: Container `overflow-hidden` clips content

The parent container at line 271 (`w-full h-screen... overflow-hidden`) clips anything exceeding 100vh. Combined with the fixed footer occupying bottom space, content that should be visible gets cut off.

---

## Solution

### Changes to `src/components/home/ClusterLayout.tsx`

#### 1. Add bottom padding to the main stage section (line 291)

Add `pb-[22vh] md:pb-[22vh]` to ensure the scrollable content always has clearance before the fixed footer. This reserves space equal to the footer height (`14vh`/`16vh`) plus its bottom offset (`6vh`/`10vh` equivalent).

```diff
- className="absolute top-[16vh] md:top-[32vh] left-0 w-full h-auto md:h-[44vh] flex flex-col md:flex-row items-center md:items-start justify-center gap-6 md:gap-[4vw] px-4 md:px-[5vw] overflow-y-auto overflow-x-hidden md:overflow-hidden custom-scrollbar z-10"
+ className="absolute top-[16vh] md:top-[32vh] left-0 w-full h-auto md:h-auto md:max-h-[calc(100vh-32vh-22vh)] flex flex-col md:flex-row items-center md:items-start justify-center gap-6 md:gap-[4vw] px-4 md:px-[5vw] overflow-y-auto overflow-x-hidden custom-scrollbar z-10 pb-[22vh] md:pb-[22vh]"
```

**Changes explained:**

| Change | Reason |
|---|---|
| `md:h-auto md:max-h-[calc(100vh-32vh-22vh)]` | Replaces hardcoded `md:h-[44vh]`. Dynamically computes available height: viewport minus top offset (32vh) minus footer space (22vh). Prevents content from growing behind the footer on any screen size. |
| `md:overflow-hidden` → removed | With `max-h` capping the height, `overflow-hidden` is no longer needed. Let `overflow-y-auto` handle vertical overflow consistently across breakpoints. |
| `pb-[22vh] md:pb-[22vh]` | Bottom padding matching the footer's vertical footprint. On mobile, when content scrolls to the bottom, the last item has breathing room above the fixed footer. On desktop, the `max-h` calculation already includes this offset, but the padding adds an extra safety margin inside the scrollable area. |

#### 2. Ensure the footer's background doesn't visually bleed (line 376)

The footer row itself is transparent (no background). If content scrolls behind it, items are visible underneath. This is by design (the footer is "floating" thumbnails), but for clarity, verify the footer containers have no unintended backgrounds that would crop visual elements:

```tsx
// Line 376-418 — no changes needed. The footer thumbnails have their own bg-[#111].
// The parent row is correctly pointer-events-none with pointer-events-auto only on thumbnails.
```

#### 3. Remove the outer `overflow-hidden` (line 271) — optional safety measure

The `overflow-hidden` on the root container can mask scrollbar issues. If the `max-h` + `overflow-y-auto` approach on the main stage works correctly, this outer clip is redundant. However, if the overflow-hidden is intentional for horizontal overflow prevention, keep it and rely on the inner scroll.

**Decision**: Keep `overflow-hidden` on the root container — it prevents horizontal scroll on the eye canvas and background elements. The inner `overflow-y-auto` on the main stage handles vertical scrolling.

---

## Effect After Fix

- **Mobile**: Cluster cards scroll vertically with the last card safely above the footer. The `pb-[22vh]` pushes the last content item well above the footer's visual footprint.
- **Desktop**: The `max-h-[calc(100vh-32vh-22vh)]` ensures the main stage never extends behind the footer. Content that exceeds this height scrolls inside the container without ever touching the footer zone.
- **All viewports**: The footer remains `fixed` at the bottom, always visible and interactive, with guaranteed separation from cluster content.
